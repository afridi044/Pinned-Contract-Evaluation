\section{Introduction}
\label{sec:introduction}

PHP still powers a large share of production web software, including many long lived legacy systems.\cite{w3techs_php,w3techs_wordpress,wordpress}
Migrating these systems across PHP versions is difficult.
Required changes are often distributed across the codebase and reflect shifts in language semantics, standard library behavior, and typing rules.\cite{php83spec,persson2015php}
Developers therefore need more than a final success signal.
They need to know which upgrade requirements were satisfied, which remain unresolved, and where residual migration work is concentrated.

Large language models are now widely used to assist software modernization.
Yet their evaluation remains coarse.
Most prior work relies on parsing success, linting, tests, or build outcomes.\cite{openai2023gpt4,roziere2023codellama,cassano2024canitedit}
These signals are useful, but they do not expose migration progress at the level of specific upgrade requirements.
They show whether an output appears acceptable.
They do not show which obligations were discharged, which persisted, or which new ones were introduced.
This makes controlled comparison difficult, especially when behavior varies with prompt design, context limits, and input size.\cite{liu2024lost,du2025context,jimenez2024swebench}

We introduce \textbf{Pinned Contract Evaluation (PCE)}, a policy grounded framework for evaluating automated code modernization.
PCE treats a pinned static analysis upgrade pipeline as an explicit migration contract.
Executing this contract on the original program induces an obligation space from the triggered upgrade rule types.
Executing the same contract on a migrated output reveals which baseline obligations were discharged, which remain, and which new obligations appear.
Because the policy is fixed and versioned, results are directly comparable across systems and experiments.
PCE therefore measures migration progress under an explicit upgrade policy.
It does not by itself establish behavioral equivalence or full semantic correctness.
Its role is different.
It provides a fine grained and reproducible measurement layer between coarse end state checks and full semantic validation.

To contextualize contract level progress, we report additional diagnostics.
These include syntax validity, structural preservation, compatibility analysis with PHPCompatibility, and a loadability sentinel for regression detection.

We instantiate PCE for PHP version migration using a pinned Rector upgrade pipeline.\cite{rector}
We evaluate five LLM systems on a 100 file WordPress 4.0 benchmark constructed to preserve diverse contract induced obligations.
Under the primary weighted all files score, obligation discharge ranges from 30.53\% to 72.80\%.
On analyzable files only, mean discharge ranges from 34.21\% to 77.11\%.
Residual work concentrates in PHP 8 rule families, and strong contract progress does not necessarily imply syntax or loadability robustness.
These findings show that policy grounded evaluation reveals migration progress and failure modes that coarse pass fail metrics cannot localize.

\paragraph*{Contributions}
This paper makes three contributions.

\begin{enumerate}

\item \textbf{Pinned Contract Evaluation.}
We present a reproducible evaluation framework that models modernization as progress through a contract induced obligation space defined by a pinned static analysis policy.

\item \textbf{A coverage oriented migration benchmark.}
We construct a 100 file WordPress 4.0 benchmark that preserves diverse obligations induced by the pinned contract and supports diagnostic evaluation.

\item \textbf{A reproducible evaluation harness.}
We provide a standardized migration and scoring pipeline with deterministic extraction, reconstruction, and fully logged artifacts.

\end{enumerate}

\subsection{Research Questions}
\label{sec:rqs}

Under a fixed contract, we ask:

\begin{enumerate}

\item \textbf{RQ1.} How do models compare in obligation discharge and residual obligation profiles?

\item \textbf{RQ2.} Which rules and language milestones remain most resistant to automated migration?

\item \textbf{RQ3.} How does contract progress relate to syntax validity, structural preservation, and loadability regressions?

\item \textbf{RQ4.} How do PCE outcomes align with results from an independent compatibility analysis using PHPCompatibility?

\end{enumerate}
