\begin{abstract}

Evaluating LLM assisted code migration remains difficult. Parsing, linting, tests, and successful builds are coarse end state checks. They do not show which upgrade requirements were actually satisfied.

We introduce Pinned Contract Evaluation (PCE), a policy grounded framework for evaluating migration progress. PCE treats a pinned static analysis upgrade pipeline as an explicit migration contract. Executing the contract on the source program induces upgrade obligations from the triggered diagnostic rules. A migrated output is scored by the fraction of these obligations it resolves under the same contract. Reductions in dry run patch suggestions provide a secondary measure of remaining migration work.

We instantiate PCE for PHP version migration with a pinned Rector toolchain and evaluate five LLM systems on a 100 file WordPress 4.0 benchmark spanning PHP 5.2 to 8.3. The benchmark preserves all 48 upgrade rules and yields 511 baseline \texttt{Rector\textbackslash Php*} file rule incidences for primary scoring. Under a unified all files protocol that assigns zero discharge to non analyzable files, weighted obligation discharge ranges from 30.53\% to 72.80\%. Residual work concentrates in PHP 8.x rule families and in a small set of difficult files.

Our results show that policy grounded evaluation exposes migration progress and failure modes that coarse pass fail metrics cannot attribute. PCE supports controlled, reproducible, and fine grained comparison of automated modernization systems.

\end{abstract}

\section{Threats to Validity}
\label{sec:threats}

\paragraph*{Construct validity}\par
PCE measures migration progress under a fixed static-analysis policy, here a pinned Rector configuration. Our main metric is file-level rule-type incidence. It is reproducible, but it does not capture within-file frequency or severity. A rule counts as resolved only when it no longer triggers in that file.

Patch-volume closure is a secondary proxy. It is based on dry-run changed lines, which can vary with formatting or restructuring. It is not a direct measure of developer effort. We reduce this risk by using the same pinned contract for all systems and by avoiding extra repair or formatting passes.

Contract scores can also be inflated by omission or degenerate rewrites. We therefore do not interpret discharge alone. We report syntax, preservation, introduced-rule regressions, a loadability sentinel, and PHPCompatibility. PCE supports broader assessment, but it does not establish semantic correctness.

\paragraph*{Internal validity}
Results may be affected by the migration harness. Context limits, chunk boundaries, extraction failures, and provider APIs can change outputs. We reduce this risk by fixing prompts, chunking, reconstruction, and decoding settings where possible. We also surface failures instead of repairing them. This makes the evaluation stricter and preserves attribution.

\paragraph*{External validity}
We study PHP migration on a WordPress-derived benchmark. The benchmark is coverage-oriented under the pinned policy. It is not frequency-representative of WordPress in the wild. Our results therefore reflect this contract and this sampling strategy. Other projects or pinned pipelines may produce different obligation spaces and difficulty profiles.

PCE is not specific to PHP or Rector. The same idea can apply wherever a pinned rule-based analysis defines migration obligations. The rule set would change, but the evaluation logic would remain the same. Empirical outcomes may still differ across ecosystems.

\paragraph*{Reliability and reproducibility}
Tool versions and configurations are pinned, and all artifacts are logged. This supports harness-level reproducibility. Still, LLM outputs may vary because of nondeterminism, missing seeds, and provider-side updates. The loadability sentinel is also environment-dependent. We therefore treat it as a directional regression signal under a fixed harness.

\section{Conclusion and Future Work}
\label{sec:conclusion}

We introduced \emph{Pinned Contract Evaluation (PCE)}, a policy-grounded framework for evaluating automated code modernization. PCE treats a pinned static-analysis pipeline as an explicit upgrade contract. It measures progress in the obligation space induced by that contract. This gives a fine-grained and reproducible view of migration progress beyond coarse end-state checks.

We instantiated PCE for PHP migration with a pinned Rector pipeline and evaluated five LLM systems on a coverage-oriented WordPress-derived benchmark. Results show clear differences in obligation discharge across systems. Later PHP obligations are harder than earlier ones. Strong contract progress does not guarantee syntax or loadability robustness. Agreement across instruments is useful, but incomplete.

The main contribution of this paper is methodological. We do not propose a new migration system. We propose a reproducible way to measure migration progress under an explicit upgrade policy. The PHP case study shows that this layer can expose residual work, failure modes, and tradeoffs that coarse pass/fail metrics miss.

\paragraph*{Future work}
We plan four extensions. First, we will study alternative pinned pipelines to test sensitivity to policy choice. Second, we will apply the framework to other ecosystems and codebases. Third, we will add stronger execution-based validation when realistic harnesses are available. Fourth, we will study feedback-driven post-editing guided by residual obligation profiles.
